# Evaluation

로봇·VLA 시스템의 **평가 방법**과 **벤치마크** — 정책(policy)이 실제로 작동하는지 측정하는 방법.

---

## 로보틱스에서 평가가 어려운 이유

이미지 분류(정답 하나)와 달리, 로봇 태스크는 다음 특성을 갖습니다:
- **연속적인 액션 공간** — 매 타임스텝마다 올바른 힘·속도·자세를 예측해야 함
- **오류 누적** — 스텝당 2% 오류가 100스텝 이후 치명적 실패로 이어짐
- **환경 변동성** — 조명, 오브젝트 배치, 표면 마찰이 달라지면 오프라인 결과와 실세계 성능이 크게 다름
- **안전이 1순위 지표** — 위험하게 태스크를 완료한 정책은 성공이 아님

핵심 과제: **오프라인 평가는 빠르고 저렴하지만 오해의 소지가 있고; 실세계 평가는 느리고 비싸지만 유일한 진실의 근거.**

---

## 평가 방법

### 1. 오프라인 평가 (시뮬 / 로그 기반)

하드웨어 없이 사전 기록된 trajectory나 시뮬레이터에서 정책을 실행합니다.

| 방법 | 측정 대상 | 사용 시점 |
|------|-----------|-----------|
| 로그 리플레이 | 기록된 참조 trajectory 대비 액션 예측 오차 | 학습 중 회귀 검사 |
| 시뮬 롤아웃 | 물리 시뮬레이터에서 태스크 완료율 | 빠른 반복, 하이퍼파라미터 튜닝 |
| BC 손실(loss) | 검증 세트에서의 평균 L1/L2 손실 | 건전성 확인; 실세계 성공률 예측 불가 |

**한계**: 로그 데이터와 실시간 실행 사이의 분포 이동(distribution shift), 물리 시뮬레이터의 접촉·마찰 격차, 실제 하드웨어 피드백 없음.

### 2. 실세계 평가

학습된 정책을 실제 하드웨어에 배포하고 결과를 측정합니다.

| 방법 | 측정 대상 | 비고 |
|------|-----------|------|
| 태스크 성공률 | 로봇이 전체 태스크를 완료했는가? | 주요 지표; 이진 또는 부분 점수 |
| Human-in-the-loop 검증 | trajectory가 안전하고 인간 관찰자에게 허용 가능한가? | 바이매뉴얼·접촉 집약 태스크에 필수 |
| Closed-loop 롤아웃 | 여러 개선 라운드에서 성공률이 어떻게 변하는가? | 단순 스냅샷이 아닌 학습 속도를 포착 |
| 실패 원인 분석 | 실패 시도의 원인 분류 | 데이터 전략 피드백으로 활용 |

### 3. 안전 통합 평가

성공만으로는 충분하지 않습니다. 모든 평가에서 함께 추적해야 할 지표:

- **Safety rate** — 안전 사고 없이 완료된 시도 비율 (충돌·과도한 힘·물체 낙하 피해 없음)
- **Safe success rate** — 태스크 완료 AND 안전한 시도의 비율 (실제 제품 지표)
- **Recovery rate** — 거의 실패할 뻔한 상황에서 인간 개입 없이 자가 복구한 비율

---

## 주요 지표

| 지표 | 정의 | 범위 |
|------|------|------|
| Task success rate | 전체 태스크 목표가 달성된 시도 비율 | 0–100% |
| Safe success rate | 성공 AND 안전 위반 없음 | 0–100% |
| Trajectory error | 예측된 엔드이펙터 경로와 참조 경로의 평균 거리 | mm (낮을수록 좋음) |
| Action prediction error | 액션 차원의 평균 L1 오차 (오프라인) | 낮을수록 좋음 |
| Steps to success | 태스크 완료까지 걸린 스텝 수 중앙값 | 낮을수록 좋음 |
| Failure mode 분포 | 분류: 파지 실패 / 배치 실패 / 협조 실패 / 안전 정지 | — |

## 벤치마크 목록

VLA·로봇 학습 커뮤니티에서 주로 사용하는 벤치마크:

| 벤치마크 | 연도 | 초점 | 규모 | 환경 | 비고 |
|----------|------|------|------|------|------|
| **RLBench** | 2020 | 테이블탑 100가지 태스크, 멀티뷰 | 태스크 100개, 각 200 demos | 시뮬 (CoppeliaSim) | 표준 기준선; 순수 시뮬 |
| **LIBERO** | 2023 | 지속 학습(lifelong), 4개 suite | suite당 10 태스크, ~50 demos | 시뮬 + 실세계 전이 | 연속 적응 테스트; 바이매뉴얼 호환 |
| **BridgeV2** | 2023 | 주방·테이블탑 실세계 다양성 | 13k+ 에피소드, 24개 환경 | 실세계 | 멀티태스크·멀티씬; 전이 벤치마크 |
| **DROID** | 2024 | 대규모 크로스-엠보디먼트 | 76k demos, 86종 로봇, 564 태스크 | 실세계 | 최대 실세계 조작 데이터셋; OXE 상위 집합 |
| **Open X-Embodiment (OXE)** | 2023 | 크로스-엠보디먼트 일반화 | 970k+ trajectory, 22종 로봇 | 혼합 | OpenVLA 학습 기반 |

### 벤치마크 활용 가이드

- **모델 개발 중**: RLBench·LIBERO로 빠른 반복 (하드웨어 불필요)
- **실세계 배포 전**: BridgeV2 스타일 분포 외 데이터(OOD) 평가로 일반화 검증
- **연구 결과 보고 시**: LIBERO·BridgeV2가 가장 많이 인용됨; 비교 가능성을 위해 둘 다 포함
- **데이터 전략 결정 시**: DROID·OXE로 대규모 데이터 다양성 기준 파악

### 오프라인 vs. 실세계 성능 격차

흔한 함정: LIBERO 시뮬에서 90%를 달성한 모델이 실세계에서 40–60%에 그칠 수 있습니다. 격차의 원인:

1. **물리 충실도** — 시뮬 접촉 모델이 실제 마찰·유연성과 다름
2. **시각 분포 이동** — 실제 조명·반사·잡동사니가 시뮬 텍스처와 다름
3. **변형 가능한 오브젝트** — 천·봉투·음식은 시뮬이 잘 안 됨
4. **시간적 정밀도** — 실제 하드웨어에는 레이턴시·지터가 있지만 시뮬은 무시

Config의 접근: 시뮬 벤치마크보다 실세계 closed-loop 평가 우선. 시뮬은 빠른 반복에 사용하지만, 배포 결정은 실물 롤아웃 결과 기반.

## Config의 평가 방식

기술 프리뷰의 "closing the loop" 전략에 맞춘 평가 흐름:

1. 학습 중 **오프라인 회귀** (액션 예측 손실, 시뮬 롤아웃)
2. 기준선 성공률 수립을 위한 **초기 하드웨어 테스트** (10–20회 시도)
3. **실패 원인 분석** — 각 실패를 분류(파지·배치·협조·안전)해 데이터 전략 개선에 반영
4. **반복적 closed-loop** — 데이터 수집·파인튜닝 라운드마다 하드웨어 재평가; 개선 곡선 추적
5. 최종 배포 게이트로 **Safe success rate** 사용 — 태스크 성공률만이 아닌
