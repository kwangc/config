# 추론 및 실시간 제어

물리 하드웨어에 VLA를 배포하려면 하드 실시간 레이턴시 제약을 충족해야 합니다. 이 페이지는 레이턴시 소스, 최적화 기법, 하드웨어 선택, 안전 통합, 그리고 배포 체크리스트를 다룹니다.

## 제어 루프 요구 사항

로봇 조작은 특정 제어 주파수에서 동작합니다. VLA는 제어 주기 내에 액션을 생성해야 합니다. 그렇지 않으면 로봇의 저수준 컨트롤러가 마지막 명령을 유지합니다.

| 태스크 유형 | 일반적 제어 주파수 | 스텝당 레이턴시 예산 | 비고 |
|-----------|-----------------|-----------------|------|
| 느린 조작 (정렬, 배치) | 5-10 Hz | 100-200 ms | 여유로운 예산; 대형 모델 가능 |
| 표준 조작 | 10-25 Hz | 40-100 ms | 일반적인 VLA 동작 포인트 |
| 접촉 풍부한 조작 | 25-50 Hz | 20-40 ms | 빠른 모델 또는 청킹 분할 상환 필요 |
| 동적 조작 | 50-200 Hz | 5-20 ms | 전체 주파수에서 VLA 적합하지 않음; 잔류 제어 사용 |
| 연속 동작 | 1 kHz | 1 ms | VLA 영역 아님; 저수준 PD 제어 |

**10 Hz 동작 포인트.** 대부분의 현재 VLA 배포는 저수준 컨트롤러가 더 빠르게 실행되더라도 정책 호출 주파수를 5-15 Hz로 운영합니다. 액션 청킹이 이것을 가능하게 합니다: VLA가 5 Hz에서 50개 액션을 생성하고, 저수준 컨트롤러가 50 Hz에서 실행합니다.

**핵심 공식:**

```
로봇 스텝당 유효 레이턴시 = 모델 레이턴시 / 청크 크기

예: model_latency=200ms, chunk_size=50Hz에서 20 스텝
  유효 레이턴시 = 200ms / 20 = 로봇 스텝당 10ms
  로봇이 동등하게 100 Hz에서 실행 가능
```

## 레이턴시 소스

| 구성 요소 | 일반적 레이턴시 | 비고 |
|---------|--------------|------|
| 이미지 캡처 + 전송 | 1-5 ms | USB/이더넷 카메라 |
| 이미지 전처리 (리사이즈, 정규화) | 0.5-2 ms | GPU 가속 |
| 비전 인코더 (ViT-B, 224×224) | 5-15 ms | GPU에서 FP16 |
| 비전 인코더 (ViT-L, 224×224) | 15-40 ms | GPU에서 FP16 |
| LLM 백본 (7B, 단일 순방향 패스) | 50-150 ms | A100/H100에서 FP16 |
| LLM 백본 (7B) | 100-300 ms | 3090/4090에서 FP16 |
| MLP 액션 헤드 | <1 ms | 무시 가능 |
| 디퓨전 헤드 (DDPM, 100 스텝) | 100-500 ms | 노이즈 제거 네트워크 크기에 따라 |
| 디퓨전 헤드 (DDIM, 10 스텝) | 10-50 ms | 훨씬 실용적 |
| 플로우 매칭 헤드 (10 스텝) | 10-50 ms | DDIM과 유사 |
| 자기회귀 디코딩 (7 액션 토큰) | 50×7 = 350 ms | LLM 순방향 패스 7회 순차적 |

현대 LLM 백본 VLA에서 두 가지 가장 큰 레이턴시 소스:
1. LLM 백본 순방향 패스
2. 액션 헤드 생성 (특히 자기회귀 또는 느린 디퓨전)

## 추론 최적화

### 액션 청킹 (분할 상환)

가장 영향력 있는 최적화입니다. 모델이 호출당 200ms 걸리고 K=20 스텝을 예측하면 스텝당 유효 비용이 10ms입니다:

```
유효 비용 = 모델 레이턴시 / K
```

모델 변경 없이 가능합니다; 액션 헤드 출력을 1 스텝에서 K 스텝으로 확장하면 됩니다. [07 — 액션 청킹](./07-action-chunking/) 참조.

### Flash Attention

타일에서 어텐션을 처리하여 전체 N×N 어텐션 행렬을 구체화하지 않는 융합 어텐션 연산:

```
표준 어텐션: O(N^2) 메모리, O(N^2) 시간
Flash Attention: O(N) 메모리, O(N^2) 연산이지만 ~2-4배 낮은 벽시계 시간
```

PyTorch 2.0+ 및 대부분의 현대 LLM 추론 프레임워크에서 지원됩니다.

### TensorRT / Torch Compile

모델 그래프를 최적화된 GPU 커널 계획으로 컴파일합니다:
- **TensorRT:** NVIDIA의 추론 최적화기; 연산자 융합, INT8/INT4 레이어 보정, 커스텀 커널 지원
- **torch.compile():** PyTorch의 JIT 컴파일; 사용하기 더 쉬움; 일반적으로 1.5-3× 속도 향상

비전 인코더와 액션 헤드(고정 크기 연산)에 가장 효과적입니다. 가변 시퀀스 길이 때문에 LLM 백본의 이점이 적습니다.

### CUDA 그래프 캡처

고정 크기 연산 (모든 호출에서 동일한 배치 크기, 동일한 시퀀스 길이)의 경우, CUDA 커널 시퀀스를 그래프로 캡처하고 커널 재시작 없이 재실행합니다:

```python
# 캡처
g = torch.cuda.CUDAGraph()
with torch.cuda.graph(g):
    output = model(static_input)

# 재실행 (커널 시작 오버헤드 없음)
g.replay()
```

스텝당 오버헤드를 1-3 ms 감소시킵니다. 높은 제어 주파수에서 의미 있습니다.

### vLLM / 연속 배칭

단일 추론 서버를 공유하는 여러 로봇의 경우: vLLM의 연속 배칭이 가변 시퀀스 길이를 가진 요청에 걸쳐 효율적인 배칭 추론을 가능하게 합니다. 하나의 GPU에 여러 팔이 있는 실험실 설정에 유용합니다.

### 투기적 디코딩 (Speculative Decoding)

자기회귀 액션 토큰 생성의 경우: 더 작은 드래프트 모델을 사용해 K개 토큰을 한 번에 제안하고, 단일 순방향 패스로 전체 모델로 검증합니다. 유효 디코딩 시간을 2-3× 줄일 수 있습니다.

## 양자화(Quantization)

수치 정밀도를 줄여 메모리와 연산 요구 사항을 줄입니다.

| 정밀도 | FP32 대비 속도 향상 | 메모리 감소 | VLA에서 품질 영향 |
|-------|-----------------|-----------|----------------|
| BF16 / FP16 | ~2× | 2× | 최소; 표준 관행 |
| INT8 | ~3-4× | 4× | 작음; 대부분의 태스크에 수용 가능 |
| INT4 | ~5-8× | 8× | 보통; 세밀한 태스크에서 정밀도 손실 |
| INT2 | >8× | 16× | 높음; VLA에 권장하지 않음 |

**구성 요소를 별도로 양자화하세요:**

- **비전 인코더:** ViT는 INT8 양자화에 상대적으로 강건합니다; 의미론적 특성이 보존됩니다
- **LLM 백본:** 7B 모델이 INT4 (bitsandbytes, GPTQ)로 조작 태스크에서 수용 가능한 품질 손실로 실행됩니다; 정밀도 중요 태스크에는 INT8이 더 안전합니다
- **액션 헤드 (디퓨전/플로우):** 백본보다 정밀도에 더 민감합니다; 가능하면 FP16 유지; 텐서별 보정이 있는 INT8은 수용 가능합니다; 연속 액션 출력에는 INT4를 피하세요

**사후 학습 양자화(PTQ) 파이프라인:**

```
1. FP32/BF16 모델 가중치 로드
2. 대표적인 로봇 관찰로 보정 (100-1000 샘플)
3. 보정된 스케일로 INT8 또는 INT4 양자화 적용
4. 조작 검증 세트에서 평가
5. 품질 손실 > 임계값이면 혼합 정밀도 사용 (INT4 백본, FP16 액션 헤드)
```

## 안전 게이트 통합

**VLA 출력을 로봇 하드웨어에 직접 연결하지 마세요.**

모든 VLA 배포에는 모델 출력과 하드웨어 명령 사이에 안전 게이트가 있어야 합니다:

```
VLA 출력: raw_action_vector
             |
     [안전 게이트]
             |
     safe_action_vector → 로봇 하드웨어

안전 게이트 확인 (순서대로):
  1. 작업 공간 경계: EEF 목표를 작업 공간 외형에 클립
  2. 관절 한계: 관절 목표를 하드웨어 한계에 클립
  3. 속도 한계: 관절당 속도를 max_velocity로 클립
  4. 가속 한계: 델타 액션을 max_acceleration으로 클립
  5. 충돌 확인 (작업 공간 모델 있는 경우): 알려진 장애물과 교차하는 액션 거부
  6. 타임아웃: 모델이 budget_ms 내에 응답하지 않으면 → hold_position 명령
  7. 워치독: 로봇이 2× control_period 내에 명령을 받지 못하면 → 비상 정지
```

**타임아웃 확인이 중요합니다.** 모델이 500ms 걸리지만 제어 주기가 100ms라면, 다음 액션이 도착하기 전 400ms 동안 로봇이 무언가를 해야 합니다. 옵션:
- **위치 홀드:** 마지막 명령된 위치를 반복; 안전하지만 정적
- **제로 속도로 감쇠:** 천천히 감속; 부드럽지만 접촉 중에 안전하지 않을 수 있음
- **다음 청크 스텝 실행:** 청킹을 사용하는 경우, 다음 청크 스텝이 이미 사용 가능함; 실행

## 배포 체크리스트

물리 하드웨어에서 VLA를 실행하기 전:

**레이턴시:**
- [ ] 목표 하드웨어에서 엔드투엔드 레이턴시 (카메라 → 액션) 측정
- [ ] model_latency / chunk_size가 제어 주파수 요구 사항을 충족하는지 확인
- [ ] 양자화와 TorchCompile 적용; 레이턴시 재측정

**정확도:**
- [ ] 액션 정규화 검증: 정규화 → 역정규화 라운드트립이 항등함수인지 확인
- [ ] 10-20개의 짧은 시험 실행; 실험실 기준선 대비 성공률 측정
- [ ] 새로운 물체 위치로 테스트하고 예상되는 일반화 행동 확인

**안전:**
- [ ] 안전 게이트 구현 및 합성 적대적 액션 입력으로 테스트
- [ ] 현재 로봇 설정에 대해 작업 공간 경계 구성 및 확인
- [ ] 비상 정지 접근 가능 및 테스트
- [ ] 타임아웃 행동 확인: 모델 행 → 위치 홀드가 올바르게 트리거

**모니터링:**
- [ ] 타임스탬프와 함께 모든 관찰, 예측된 액션, 실행된 액션 로그
- [ ] 안전 게이트 개입 로그 (얼마나 자주, 왜)
- [ ] 배포 후 분석을 위해 시험당 성공/실패 결과 로그
- [ ] 이상 감지 알림 설정 (비정상적인 액션 크기, 높은 불확실성)

**롤백:**
- [ ] 이전 정책 체크포인트 보유 및 5분 내 배포 가능
- [ ] 실패 카테고리에 대한 폴백 행동 정의 (예: 파지 실패 → 홈 포즈로 복귀)

## 생각해볼 거리

- VLA의 엔드투엔드 레이턴시가 이론적 예산을 충족하지만 로봇이 여전히 저키하게 동작한다면, 문제는 종종 평균 레이턴시가 아닌 지터(레이턴시 분산)입니다 — p50이 아닌 p99 레이턴시를 측정하고, 최악의 경우 급증을 흡수하기 위해 작은 액션 버퍼를 추가하세요.
- INT4 양자화가 6× 속도 향상을 제공하지만 로봇이 목표를 지나치기 시작한다면, 액션 헤드가 범인입니다 (연속 출력이 토큰 분류보다 정밀도에 더 민감) — 최선의 속도-정밀도 트레이드오프를 위해 백본을 INT4로 양자화하되 액션 헤드는 FP16으로 유지하세요.
