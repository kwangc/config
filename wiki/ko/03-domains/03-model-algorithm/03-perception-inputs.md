# VLA를 위한 인식 입력

인식 입력의 품질과 다양성은 VLA 성능에 가장 큰 영향을 미치는 요소 중 하나입니다. 이 페이지는 현대 VLA 시스템에서 사용되는 센서 유형, 토크나이제이션 전략, 그리고 멀티카메라 융합 방법을 다룹니다.

## 이미지 입력

### 카메라 종류와 배치

| 카메라 종류 | 배치 | 일반적 용도 | 장점 | 단점 |
|------------|------|------------|------|------|
| 제3자 시점(third-person) / 오버헤드 | 작업 공간 위에 고정 | 장면 개요, 물체 위치 파악 | 넓은 시야, 안정적 | 엔드이펙터에서 멀리 있음, 팔에 의한 가림 |
| 손목 카메라(wrist camera) | 엔드이펙터에 장착 | 근거리 조작, 접촉 | 액션 현장의 고해상도 | 팔과 함께 움직임(불안정한 시점), 장면 맥락 제한 |
| 자아 중심(egocentric) / eye-in-hand | 로봇 머리 / 손 장착 | 내비게이션 + 조작 결합 | 자연스러운 인간 시점 | 팔과의 시차(parallax), 보정 필요 |
| 측면 카메라 | 측면에 고정 | 깊이 추정, 파지 각도 | 보완적 각도 | 추가 보정, 더 많은 토큰 |

대부분의 조작 VLA는 **최소 두 개의 카메라**를 사용합니다: 장면 맥락을 위한 오버헤드/제3자 시점 카메라와 접촉 수준의 정밀도를 위한 손목 카메라. 손목 카메라는 파지가 성공하고 있는지 판단하는 데 종종 가장 정보가 풍부한 단일 시점입니다.

### 일반적 해상도

- **224×224**: 표준 ViT 입력; 대부분의 VLA 파이프라인에서 사용(RT-2, OpenVLA, π0); 빠름; 피킹과 플레이싱에 충분
- **256×256 또는 320×240**: 약간 더 높음; 세밀한 파지 세부 정보가 중요할 때 사용
- **640×480**: 전체 웹캠 해상도; ViT에 직접 입력하는 경우는 거의 없음 — 일반적으로 먼저 다운샘플링
- **더 높은 해상도(1280×720)**: ViT 전에 서브영역 크롭을 수행할 때만; 손목 카메라가 엔드이펙터 주변 224×224 영역을 크롭하는 데 사용하기도 함

더 높은 해상도는 작업이 물체의 텍스트를 읽거나 세밀한 시각적 특성을 구별해야 하는 경우가 아니면 토큰 수 증가에 비해 거의 가치가 없습니다.

## ViT 패치 토크나이제이션

Vision Transformer(ViT)는 다음 단계를 통해 이미지를 토큰 시퀀스로 변환합니다:

```
입력: H × W × C 이미지 (예: 224 × 224 × 3)

1단계: 겹치지 않는 패치로 분할
  - 패치 크기 P × P (예: P=16)
  - 패치 수: N = (H/P) * (W/P) = 196 (224×224, P=16일 때)

2단계: 각 패치를 벡터로 평탄화
  - 각 패치: P * P * C = 768 값

3단계: 임베딩 차원 d로 선형 투영
  - patch_embed: (P*P*C,) → (d,)  (d는 일반적으로 768 또는 1024)

4단계: 학습 가능한 위치 임베딩 추가

5단계: [CLS] 토큰 선두에 추가 (선택적)

6단계: L개의 트랜스포머 인코더 레이어 통과
  - 출력: N개 패치 토큰 + 선택적 CLS 토큰, 각각 차원 d
```

### 패치 크기 트레이드오프

| 패치 크기 | 패치 수 (224×224) | 토큰 시퀀스 길이 | 공간 해상도 | 연산량 |
|----------|-----------------|----------------|------------|--------|
| 32×32 | 49 | 49 | 거침 | 빠름 |
| 16×16 | 196 | 196 | 중간 | 보통 |
| 8×8 | 784 | 784 | 세밀 | P=16보다 4× 느림 |
| 4×4 | 3136 | 3136 | 매우 세밀 | 매우 느림 |

접촉 감지나 그리퍼 스케일 특성이 필요한 조작 작업에는 P=8 또는 P=14(SigLIP에서 사용)가 허용 가능한 연산 비용으로 의미 있는 공간 해상도를 제공합니다. 장면 수준 이해(물체 위치 파악, 장면 파싱)에는 P=16 또는 P=32가 일반적으로 충분합니다.

### CLS 토큰 vs 풀링

- **CLS 토큰**: 시퀀스 앞에 추가되는 학습된 토큰; 트랜스포머 레이어 후 CLS 표현이 전역 이미지 정보를 집계; 이미지 수준 태스크(분류, 언어 정렬)에 사용
- **평균 풀링**: 모든 패치 토큰 임베딩의 평균; CLS와 유사하지만 전용 학습 토큰 없음
- **전체 패치 시퀀스**: N개 패치 토큰 전체를 하위 모듈에 전달; 공간 관계가 액션 헤드에 중요할 때 필요; 현대 VLA에서 표준

VLA 액션 예측에서는 거의 항상 전체 패치 시퀀스를 사용합니다 — 액션 헤드는 물체의 위치에 대한 공간 정보가 필요하지, 단순한 전역 장면 서술자가 아니기 때문입니다.

## 깊이와 포인트 클라우드

### 깊이가 도움이 되는 경우

깊이(depth)는 명시적 3D 기하학 정보를 추가합니다: 물체가 얼마나 떨어져 있는지, 3D 형태, 표면 법선. 다음 경우에 유용합니다:

- 정밀한 수직 배치 (쌓기, 삽입)
- 가려진 물체 추정
- 곡선이나 불규칙한 물체 파지
- 2D 이미지의 깊이 모호성이 실패를 유발하는 모든 태스크

### 깊이를 건너뛰어야 할 경우

깊이 센서는 비용, 보정 요구 사항, 레이턴시를 추가합니다. 또한 모델 복잡성을 증가시키는 추가 토큰이나 채널을 생성합니다. 다음 경우에는 건너뜁니다:

- 두 개의 RGB 카메라의 스테레오 단서로 태스크 해결 가능
- 센서 비용이 제약인 경우
- 깊이 데이터에 노이즈가 많은 경우 (투명 물체, 반짝이는 표면으로 structured-light/ToF 실패)
- 로봇 작업 공간이 잘 제한된 경우 (테이블 높이를 알고 있음)

### 표현

- **밀집 깊이 맵**: RGB와 동일한 H×W 그리드, 각 픽셀에 거리 저장; 추가 채널(RGBD 입력)로 연결; 입력 투영의 경미한 수정으로 ViT와 동일하게 처리
- **포인트 클라우드**: 픽셀당 3D 좌표; RGB 토큰과 결합하기 전에 PointNet 또는 희소 3D 합성곱으로 처리
- **희소 키포인트**: 감지된 물체 키포인트에서만 깊이를 인코딩; 경량이지만 키포인트 감지 단계 필요

## 고유감각 (Proprioception)

고유감각(proprioceptive) 상태는 로봇의 내부 센서 읽기값입니다 — 현재 로봇 몸체의 구성을 정책에게 알려줍니다.

### 포함되는 신호

| 신호 | 기호 | 일반적 차원 | 설명 |
|------|------|------------|------|
| 관절 각도 | q | 팔당 6–7 | 현재 관절 위치 (라디안) |
| 관절 속도 | dq | 팔당 6–7 | 현재 관절 각속도 |
| 엔드이펙터 위치 | xyz | 3 | 엔드이펙터의 데카르트 위치 |
| 엔드이펙터 방향 | quat 또는 euler | 4 또는 3 | 엔드이펙터의 회전 |
| 그리퍼 상태 | g | 1–2 | 그리퍼 개/폐 위치 또는 힘 |
| 촉각 / 힘-토크 | ft | 6 | 손목 힘-토크 센서 (있을 때) |

6-DOF 팔과 평행 그리퍼의 경우 전체 고유감각 벡터는 약 20개 부동소수점입니다.

### 고유감각이 중요한 이유

로봇은 카메라만으로는 자신을 신뢰성 있게 "볼" 수 없습니다. 팔이 자주 자신이나 목표 물체를 가립니다. 카메라 보정 오류가 팔 구성에 따라 축적됩니다. 가림이 없어도 픽셀에서 세밀한 운동을 예측하는 것은 매우 데이터 집약적입니다.

고유감각은 로봇 자신의 상태에 대한 노이즈 없는 기준 진실(ground truth)입니다. 고유감각에 접근할 수 있는 정책은 더 빠르게 수렴하고, 더 나은 정밀도를 달성하며, 조명과 배경 변화에 더 잘 일반화합니다.

### 고유감각 인코딩 방법

**먼저 정규화:**

```
q_norm = (q - q_mean) / q_std    # 학습셋 통계 사용
```

또는 관절 한계를 사용해 [-1, 1]로 하드 클립:

```
q_norm = 2 * (q - q_min) / (q_max - q_min) - 1
```

**주입 전략:**

1. **추가 토큰으로 선두에 추가**: MLP(proprio) → 차원 d의 임베딩, 토큰 시퀀스에 선두로 추가; 트랜스포머가 입력 맥락의 일부로 상태 토큰에 어텐션; 현대 VLA에서 가장 일반적 (π0, ACT)
2. **CLS 임베딩에 추가**: MLP(proprio)를 CLS 임베딩에 더함; 경량이지만 표현력 낮음 — 정보가 강한 시각적, 언어적 특성 속에 섞임
3. **관찰 벡터에 연결**: 모든 토큰을 평탄화하고, proprio를 연결하고, 액션 헤드 MLP에 입력; 소형 모델에서만 가능

전략 1(토큰으로 선두에 추가)이 현대 VLA에서 가장 일반적입니다.

## 손목 카메라 특수 사례

손목 카메라(wrist camera)는 로봇 엔드이펙터 위 또는 근처에 장착됩니다. 근거리, 그리퍼 중심 시점을 제공합니다.

- **팔과 함께 움직임**: 고정 카메라와 달리, 팔이 움직이면서 시점이 변합니다; 이미지가 엔드이펙터의 접근 궤적을 직접 반영합니다
- **그리퍼 상태와 높은 상관관계**: 손목 카메라에서 보이는 것은 정확히 그리퍼가 상호작용하는 것입니다; 접촉 수준 태스크(플러그 삽입, 작은 물체 파지)에 가장 정보가 풍부한 시점
- **근거리 세부 정보**: 오버헤드 카메라에서는 보이지 않는 밀리미터 규모의 특성 해상도 가능
- **레이턴시 민감성**: 카메라가 움직이기 때문에, 높은 팔 속도에서 모션 블러가 이미지 품질을 저하시킬 수 있음

양팔(bimanual) 설정에서는 각 팔에 보통 자체 손목 카메라가 있어 근거리 시점의 토큰 수가 두 배가 됩니다.

## 멀티카메라 융합

### 토큰 연결(Token Concatenation)

가장 간단한 방법: 각 카메라를 자체 ViT로 실행하고, 토큰 시퀀스를 수집하고, 모두 연결하여 백본 트랜스포머에 입력합니다.

```
tokens_cam1: N1 토큰 (예: 224×224, P=16에서 196)
tokens_cam2: N2 토큰
tokens_lang: L 토큰
tokens_prop: 1 토큰

결합: N1 + N2 + L + 1 토큰
```

작동하지만 토큰 수가 카메라 수에 선형적으로 증가하여 자기 어텐션 연산이 이차적으로 증가합니다. 3개 카메라와 50개 언어 토큰으로도 결합 시퀀스가 ~650 토큰이 됩니다.

### 카메라별 별도 ViT vs 공유 가중치

| 방식 | 파라미터 수 | 사용 시점 |
|------|-----------|----------|
| 카메라별 별도 ViT | 3× ViT 파라미터 | 카메라들의 통계적 특성이 매우 다를 때 (손목 vs 오버헤드) |
| 공유 ViT 가중치 | 1× ViT 파라미터 | 카메라들의 이미지 분포가 유사할 때; 과적합 감소; 더 일반적 |

실제로는 모든 카메라에 걸쳐 ViT 가중치를 공유하는 것(다른 위치 임베딩 또는 카메라 ID 임베딩과 함께)이 파라미터를 줄이고, 더 데이터 효율적인 학습을 가능하게 하며, 종종 별도 ViT만큼 또는 더 잘 수행합니다.

### 위치/카메라 임베딩

멀티카메라에 공유 ViT 가중치를 사용할 때, **카메라 ID 임베딩**으로 구별합니다: 백본 이전에 해당 카메라의 모든 토큰에 추가되는 학습된 벡터. 이를 통해 백본은 별도 가중치 없이도 어느 카메라의 토큰인지 알 수 있습니다.

## 생각해볼 거리

- VLA가 실험실 환경에서는 잘 일반화하지만 고객 현장에서 실패한다면, 모델이 먼저가 아닙니다 — 새로운 조명, 배경, 물체 외양이 ViT의 시각적 분포 내에 있는지 확인하세요; 손목 카메라를 추가하면 먼 장면 변화를 보상하는 불변한 근거리 시점을 자주 제공합니다.
- 깊이 채널을 추가해도 성능이 향상되지 않는다면, ViT 패치 토크나이저가 RGBD 입력을 올바르게 처리하도록 조정되지 않은 것이 가능한 이유입니다 — 깊이 값은 RGB와 매우 다른 크기와 분포를 가지므로 투영 전에 별도로 정규화해야 합니다.
- 토큰 수가 배포 병목이라면 P=16에서 P=32로 전환하면 성능 손실이 미미하면서 시퀀스 길이가 절반으로 줄어듭니다; 손목 카메라 해상도는 공간 세부 정보가 가장 중요한 곳이므로 거기서는 P를 작게 유지하세요.
- 학습 파이프라인에 고유감각이 빠져 있고 정책이 세밀한 배치에서 실패한다면, 픽셀만으로 엔드이펙터의 정확한 위치를 추론하려고 합니다 — 이는 직접 제공하는 것보다 훨씬 더 많은 데이터가 필요합니다; 카메라를 더 추가하기 전에 proprio를 추가하세요.
